## 🔍 DETALHES DOS BUGS

### BUG #1: Auto-Preenchimento de `valor_cobrado` 🔴 CRITICAL

**O que era:**
- Sistema tem tabela de preços por modelo + tipo de reparo
- Ao criar/editar OS, usuário tinha que preencher `valor_cobrado` manualmente
- Quando selecionava modelo + tipo de reparo, **NENHUMA sugestão de preço era exibida**

**O que foi corrigido:**
1. ✅ Adicionado endpoint backend `GET /api/precos/sugerir` que:
   - Recebe: `modelo`, `reparo_ids` (comma-separated), `tabela` (optional)
   - Retorna: preço total sugerido (soma de todos os reparos selecionados)
   - Acessível a todos os usuários autenticados (não apenas admin)

2. ✅ Adicionado `useEffect` em `NewOrder.jsx`:
   - Dispara quando `form.modelo` ou `selectedReparos` mudam
   - Auto-preenche `valor_cobrado` com o preço sugerido
   - Sempre sobrescreve (é uma nova OS)

3. ✅ Adicionado `useEffect` em `EditOrder.jsx`:
   - Somente dispara APÓS a OS ser carregada do DB
   - Evita sobrescrever valores já definidos na primeira renderização
   - Usa flag `initialized.current`

**Commit:** `fix: auto-preencher valor_cobrado pela tabela de preços`

### BUG #2: URL do PDF do Relatório IR Phones 🟠 HIGH

**O que era:**
- Página de Relatórios tem dois abas: "IR Phones" e "Técnicos"
- Botão "Exportar PDF" para "IR Phones" gerava URL incorreta
- Tentava acessar `/api/relatorios/pdf/irphones` (não existe)
- Backend tem rota em `/api/relatorios/pdf/ir-phones` (com hífen)

**O que foi corrigido:**
- Função `precos.pdfUrl()` em `client.js` agora mapeia:
  - `"irphones"` → `"ir-phones"` (adiciona hífen)
  - `"tecnicos"` → `"tecnicos"` (sem alteração)

**Código antes:**
```javascript
pdfUrl: (tipo, params = {}) => `${BASE}/relatorios/pdf/${tipo}...`
```

**Código depois:**
```javascript
pdfUrl: (tipo, params = {}) => {
  const qs = new URLSearchParams(params).toString();
  const endpoint = tipo === "irphones" ? "ir-phones" : tipo;
  return `${BASE}/relatorios/pdf/${endpoint}${qs ? "?" + qs : ""}`;
}
```

**Commit:** `fix: auto-preencher valor_cobrado pela tabela de preços...`

### BUG #3: `clienteHistory` URL Incorreta 🟡 MEDIUM

**O que era:**
- API client tinha função `clienteHistory(nome)` para buscar histórico do cliente
- Gerava URL: `GET /api/ordens/cliente/{nome}` (rota inexistente)
- Backend tem rota em: `GET /api/ordens/historico-cliente?cliente={nome}`
- Função nunca era usada no frontend (dead code)

**O que foi corrigido:**
- Função agora aponta para a rota correta

**Código antes:**
```javascript
clienteHistory:(nome) => get(`/ordens/cliente/${encodeURIComponent(nome)}`)
```

**Código depois:**
```javascript
clienteHistory:(nome) => get(`/ordens/historico-cliente?cliente=${encodeURIComponent(nome)}`)
```

**Impacto:**
- Pequeno: função não é usada, mas seria bug se alguém a implementasse

**Commit:** `fix: auto-preencher valor_cobrado pela tabela de preços...`

### BUG #4: Campo `cor` Não Limpo ao Trocar `modelo` 🟡 MEDIUM

**O que era:**
- Em `NewOrder.jsx`: quando usuário muda modelo, `cor` é automaticamente limpo
- Em `EditOrder.jsx`: quando usuário muda modelo, `cor` **NÃO ERA LIMPO**
- Causava possibilidade de salvar combinação inválida (ex: iPhone 13 + cor somente disponível no iPhone 14)

**Por que isso é problema:**
- iPhone 13 disponível em: Midnight, Starlight, Red, Blue, Green
- iPhone 14 disponível em: Midnight, Starlight, Dark Purple, Gold, Red
- Se ordem foi criada com iPhone 13 + Green, e depois mudou para iPhone 14
- Green é inválido para iPhone 14, mas `cor` não era limpo

**O que foi corrigido:**
- `EditOrder.jsx` modelo dropdown agora também limpa `cor`

**Código antes:**
```jsx
<Select value={form.modelo} onValueChange={(v) => setField("modelo", v)}>
```

**Código depois:**
```jsx
<Select value={form.modelo} onValueChange={(v) => { setField("modelo", v); setField("cor", ""); }}>
```

**Commit:** `fix: auto-preencher valor_cobrado pela tabela de preços...`

## 📝 NOTAS TÉCNICAS

### Decisões de Design

**1. Por que endpoint separado `/precos/sugerir` em vez de incluir na `/constantes`?**
- `/precos` está protegido por permission (admin-only originalmente)
- Novo endpoint é acessível a todos os usuários autenticados
- Mantém separação de responsabilidades
- Permite future features (ex: lookup por cliente específico)

**2. Por que não usar `NewOrder.jsx` e `EditOrder.jsx` diferente?**
- `NewOrder`: sempre sobrescreve (é nova OS, ok perder edições manuais)
- `EditOrder`: sobrescreve apenas se usuário mudou modelo/reparo (preserva dados editados)
- Uso de `useRef(initialized)` para rastrear primeira carga

**3. Por que lipar `cor` ao trocar modelo?**
- Evita combinações inválidas modelo/cor no banco de dados
- Força usuário a re-selecionar cor válida para novo modelo
- Mesma lógica já existia em `NewOrder.jsx`

### Possíveis Melhorias Futuras
- [ ] Mostrar preço sugerido visualmente (tooltip/badge) ao lado do campo
- [ ] Permitir editção de preço com "diff" visual vs. sugestão
- [ ] Cache de preços no localStorage
- [ ] Endpoint para sugerir preço por cliente específico (tabela "clientes" vs "ir_phones")
- [ ] Histórico de mudanças de preço em OS editadas
